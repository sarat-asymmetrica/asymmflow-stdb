## The Wright Brothers Moment (Lead with the Boldest Insight)

Before the structured analysis: here is the one sentence that unifies everything.

**Every entity in AsymmFlow is a financial obligation traveling through a directed acyclic graph of states, where each state transition either increases or decreases the net outstanding balance of some party.**

That is all of it. 156,000 lines of code. One sentence.

Every table is either:
1. A **party** (who owes whom — Customer, Supplier, Member)
2. An **obligation** (what is owed — Invoice, SupplierInvoice, PO)
3. A **settlement** (what was paid — Payment, SupplierPayment)
4. A **pipeline stage** (the journey from opportunity to obligation — Opportunity, Offer, Order, CostingSheet)
5. A **ledger entry** (the audit trail — ActivityEvent, FollowUp)

Everything else is derived. Every reducer either creates obligations, settles them, or advances pipeline stages toward obligation creation.

## 1. The Universal State Machine

### Algebraic Definition

Every stateful entity in AsymmFlow is an instance of a **finite automaton** over a common abstract type:

```
StateMachine(S, T, terminal: S -> Bool) where:
  S = finite set of states
  T ⊆ S × S = valid transitions (directed edges)
  terminal: S -> Bool = predicate for absorbing states
```

The concrete instances observed across all domains:

| Entity | States | Terminal States | Transition Count |
|--------|--------|-----------------|-----------------|
| Invoice | Draft, Sent, PartiallyPaid, Paid, Overdue, Cancelled | Paid, Cancelled | 7 |
| SupplierInvoice | same | same | 7 |
| PO | Draft, Approved, Sent, PartiallyReceived, Received, Cancelled | Received, Cancelled | 6 |
| Offer | Draft, Sent, Ongoing, Accepted, Rejected, Lost | Accepted, Rejected, Lost | 7 |
| Order | Created, Confirmed, InProduction, Shipped, Delivered, Cancelled | Delivered, Cancelled | 8 |
| Opportunity | Lead, Qualified, Proposal, Negotiation, ClosedWon, ClosedLost | ClosedWon, ClosedLost | 8 |

**The common shape across all six**: approximately 6 states, 2 terminal states, a linear happy path, one cancellation escape, and one partial/intermediate state.

### Is this a Monoid?

Yes, with a caveat. State transitions compose:

```
apply(apply(draft, "approve"), "send") = apply(draft, "approve >> send")
```

The composition `>>` is a partial monoid: it is defined only when the target state of the left transition matches the source state of the right transition. The identity element is the "stay" transition (no-op). Associativity holds. This is exactly the category of paths in a directed graph — a **free category** generated by the transition graph.

### The Universal Generic Reducer

The entire state machine enforcement across all six entity types collapses to one parameterized function:

```typescript
// One function, six instantiations
function genericTransition<Entity extends { status: S, id: bigint }>(
  ctx: ReducerContext,
  entityId: bigint,
  newStatus: S,
  VALID_TRANSITIONS: Record<string, string[]>,
  TERMINAL_STATES: Set<string>,
  lookup: (id: bigint) => Entity | undefined,
  update: (e: Entity & { status: S, updatedAt: Timestamp }) => void,
): void {
  const entity = lookup(entityId);
  if (!entity) throw new SenderError('Entity not found');

  const current = entity.status.tag;
  if (TERMINAL_STATES.has(current)) {
    throw new SenderError(`Cannot transition from terminal state: ${current}`);
  }

  const allowed = VALID_TRANSITIONS[current] ?? [];
  if (!allowed.includes(newStatus.tag)) {
    throw new SenderError(`Invalid transition: ${current} -> ${newStatus.tag}`);
  }

  update({ ...entity, status: newStatus, updatedAt: ctx.timestamp });
}
```

Currently the codebase has six separate implementations of this exact logic, with different variable names, in six different files. The `update_po_status` reducer in the implementation plan is one instance. They are all the same function wearing different hats.

The transition tables for all six machines can be declared as data, not code:

```typescript
const TRANSITIONS = {
  invoice:     { Draft: ['Sent'], Sent: ['PartiallyPaid','Paid','Overdue','Cancelled'], ... },
  po:          { Draft: ['Approved','Cancelled'], Approved: ['Sent','Cancelled'], ... },
  offer:       { Draft: ['Sent'], Sent: ['Ongoing','Accepted','Rejected','Lost'], ... },
  order:       { Created: ['Confirmed','Cancelled'], Confirmed: ['InProduction'], ... },
  opportunity: { Lead: ['Qualified','ClosedLost'], Qualified: ['Proposal','ClosedLost'], ... },
} as const;
```

**Concrete recommendation**: One generic `transition` reducer parameterized by entity type. Every state change in the entire ERP routes through it. The valid transition tables live in a single config object. Removing them from procedural code and putting them in declarative data cuts the state machine surface area by approximately 80%.

## 2. The Category Theory Lens

### Objects and Morphisms

Model AsymmFlow as a category **Biz** where:

- **Objects**: The entity types `{Customer, Supplier, Product, Opportunity, CostingSheet, Offer, Order, Invoice, Payment, PO, SupplierInvoice, SupplierPayment}`
- **Morphisms**: Business operations (reducers) that transform one entity's state or create new entities from existing ones
- **Composition**: When the output of one operation is the required input of another

The morphisms with explicit codomain relationships:

```
create_opportunity(Customer) -> Opportunity
create_costing(Opportunity) -> CostingSheet
create_offer(CostingSheet) -> Offer
accept_offer(Offer) -> Order           [Offer.status becomes Accepted]
create_invoice(Order) -> Invoice
record_payment(Invoice) -> Payment     [Invoice.outstanding decreases]

create_po(Supplier, Order) -> PO
create_supplier_invoice(PO) -> SupplierInvoice
record_supplier_payment(SupplierInvoice) -> SupplierPayment
```

### The Pipeline as a Functor

There are two parallel pipelines. Each is a functor from a common source category:

**Customer Pipeline** (revenue):
```
Lead -> Qualified -> Proposal -> Negotiation -> ClosedWon
  = Opportunity{stage} -> Offer -> Order -> Invoice -> Payment
```

**Supplier Pipeline** (cost):
```
Order -> PO -> SupplierInvoice -> SupplierPayment
```

Both pipelines are functors `F: Timeline -> Biz` mapping time intervals to business entities. They compose to produce the **gross margin** functor:

```
Margin = CustomerPipeline - SupplierPipeline
```

The entire profitability of PH Trading is the natural transformation between these two functors.

### The Forgetful Functor and Perspectives

A "view" in STDB terms is literally a forgetful functor — it drops information from an entity to expose only what a given role needs.

| Role | Forgetful Functor Applied |
|------|--------------------------|
| Sales | Order forgets: supplierId, invoiceId, vatAmount, outstandingBhd |
| Accountant | Order forgets: competitorPresent, winProbabilityBps, ownerId |
| Operations | Invoice forgets: everything (no access) |
| Management | Identity functor — loses nothing |

This is not a metaphor. Each STDB view is literally the construction of a simpler object category from the full category. The permission system is a set of forgetful functors indexed by role.

### Composition: Where the Business Actually Lives

The interesting compositions are the **non-trivial ones** — where the business logic creates value:

1. `accept_offer ∘ create_offer ∘ create_costing ∘ create_opportunity`: The full sales funnel. Every step's output is the next step's input. Composing these five morphisms is the business converting an inquiry into a committed order.

2. `record_payment ∘ create_invoice ∘ accept_offer`: Converting a commitment into cash. The entire purpose of the company.

3. `record_supplier_payment ∘ create_supplier_invoice ∘ create_po ∘ accept_offer`: The cost side. Completing this composition is what determines the margin.

**The margin is the categorical difference between morphism chain 2 and morphism chain 3**. The business intelligence layer's entire job is to make this difference visible in real time.

### The Adjunction: Credit Terms

The most elegant structure in the whole system is an adjunction. Customer credit terms create a **time-shifted** version of payment:

```
F: Deliveries -> Receivables   (ship now, invoice now)
G: Receivables -> CashInflows  (wait 90-120 days)
```

`G ∘ F` is the cash conversion cycle. The grade system (A/B/C/D customers) is a measure of how well this adjunction behaves — Grade A customers make `G ∘ F` nearly the identity; Grade D customers make it undefined (the unit of the adjunction fails to naturalize).

## 3. The Minimum Description Length Analysis

### What Is Truly Irreducible?

Applying Kolmogorov complexity thinking: what information cannot be derived from other information?

**Cannot be derived (must be stored explicitly):**
- Customer identity and grade (external ground truth)
- Supplier identity and product catalog (external ground truth)
- The amount of each transaction (measured from the world)
- The timestamp of each event (clock reading)
- Who performed each action (identity, non-forgeable)

**Can be derived (should not be stored, currently is):**
- `outstanding_bhd` on Invoice = `total_bhd - SUM(payments where invoiceId = this.id)`
- `invoice_status` = function of outstanding_bhd and due_date
- Customer `is_credit_blocked` = function of grade and total outstanding
- Supplier `current_year_sales_bhd` = SUM(supplier invoices for current year)
- `win_probability` = function of competitor, customer grade, product type

The current schema stores several derived quantities as materialized columns and updates them on each mutation. This is a source of bugs (the Phase 18 data reconciliation was entirely caused by mismatched materialized state) and unnecessary code.

**The minimum table count analysis:**

Current proposed: 17 tables
Theoretical minimum: 10 tables

| Table | Status | Justification |
|-------|--------|---------------|
| Member | Irreducible | Identity ground truth |
| Customer | Irreducible | External entity |
| CustomerContact | Irreducible | Cannot derive contacts from customers |
| Supplier | Irreducible | External entity |
| Product | Reducible candidate | Could inline into OrderItem with supplier ref |
| Opportunity | Irreducible | Pipeline stage record |
| CostingSheet | Reducible | Could merge with Offer (one-to-one relationship) |
| Offer | Irreducible | The commitment record |
| Order | Irreducible | The fulfillment record |
| OrderItem | Irreducible | Line items cannot collapse into Order |
| PurchaseOrder | Irreducible | Supplier fulfillment record |
| Invoice | Irreducible | Financial obligation record |
| Payment | Irreducible | Settlement record |
| SupplierInvoice | Reducible | Mirror of Invoice — could use one Invoice table with `party_type` enum |
| SupplierPayment | Reducible | Mirror of Payment — same argument |
| ActivityEvent | Reducible | Can be a STDB scheduled table that self-prunes |
| FollowUp | Irreducible | Task management cannot be derived |

**Aggressive consolidation path (10 tables):**

Merge `Invoice + SupplierInvoice` into `Obligation { party_type: enum(Customer, Supplier), party_id, ... }`.
Merge `Payment + SupplierPayment` into `Settlement { obligation_id, amount, ... }`.
Merge `CostingSheet` into `Offer` as fields (revision history as an array field or separate `OfferRevision` table if needed).
Drop `Product` as a standalone table; embed product reference and unit price directly in `OrderItem` and `POItem`.

**Conservative consolidation (13 tables, recommended):** Keep the Customer/Supplier symmetry for readability but acknowledge the duplication. Merge only the obvious redundancies: CostingSheet into Offer, drop standalone Product.

### The Reducer Minimum

Current implementation plan: approximately 20 reducers
Theoretical minimum: 8 reducers

```
1. join_member(nickname) -> Creates member identity
2. upsert_party(party_type, data) -> Customer OR Supplier (one reducer, two entity types)
3. advance_pipeline(entity_type, entity_id, new_stage, payload) -> ALL state transitions
4. create_obligation(party_id, order_id, line_items) -> Invoice OR SupplierInvoice
5. record_settlement(obligation_id, amount, reference) -> Payment OR SupplierPayment
6. create_followup(entity_type, entity_id, note, due_at) -> FollowUp
7. complete_followup(followup_id) -> Marks done
8. log_event(kind, entity_type, entity_id, description) -> Internal audit trail
```

`advance_pipeline` is the universal state machine reducer. `upsert_party` handles both sides of the buy/sell relationship. `create_obligation` / `record_settlement` are the financial pair.

**The LOC reduction if this is implemented:**
- 8 reducers instead of ~20 = 60% fewer reducer functions
- All state machines consolidated = ~400 LOC instead of ~1,200 LOC
- Symmetric party handling = ~150 LOC instead of ~400 LOC
- **Total reducer layer: ~800 LOC instead of ~2,000 LOC**

## 4. The Hyperbolic Geometry Connection (Mirzakhani)

Mirzakhani proved that the number of geodesics of length at most L on a hyperbolic surface of genus g grows as `C × L^(6g-6)` where C depends on the Weil-Petersson volume of the moduli space.

The business pipeline has a natural hyperbolic geometry interpretation.

### The Sales Surface

Model the space of opportunities as a hyperbolic surface. Each opportunity is a point. The metric is determined by time-to-close and win-probability — nearby opportunities cluster by similarity.

The **geodesics** on this surface are the optimal sales paths: given an opportunity at position P, the geodesic to "Closed Won" is the shortest path through the space of sales actions that reaches that terminal state. This is precisely what the win probability predictor computes — it is measuring the geodesic distance from current opportunity state to the "Closed Won" attractor.

### Pants Decomposition as Module Decomposition

Mirzakhani's central tool was the **pants decomposition**: cutting a surface along disjoint simple closed curves to obtain a collection of three-holed spheres ("pairs of pants"). Each pair of pants is the simplest possible connected surface with boundary.

AsymmFlow decomposes naturally into pants:

**Pants 1**: Customer <-> Opportunity <-> Offer
(Three boundary circles: new leads in, offers out, lost deals out)

**Pants 2**: Offer <-> Order <-> Invoice
(Three boundary circles: accepted offers in, invoices out, cancelled orders out)

**Pants 3**: Supplier <-> PO <-> SupplierInvoice
(Three boundary circles: linked orders in, supplier payments out, cancelled POs out)

**Pants 4**: Invoice <-> Payment <-> Cash
(Three boundary circles: invoices in, settled obligations out, overdue receivables out)

Each pair of pants is an independent module. They connect at the boundaries: an `Offer` accepted in Pants 1 becomes the input to Pants 2. A `PO` created in Pants 2 becomes the input to Pants 3.

This is not just aesthetic. The pants decomposition tells us precisely which modules can be developed and tested independently (each pair of pants is self-contained) and where the integration points are (the boundary curves).

**The genus of the business graph**: Count the independent cycles. There are two: the customer cash cycle (sell -> invoice -> collect) and the supplier cost cycle (order -> pay -> receive). The overall graph has genus 2 — two independent loops. This means the system can deadlock in exactly two ways: customer doesn't pay (cycle 1 breaks), or supplier doesn't deliver (cycle 2 breaks). The entire risk management of PH Trading is about keeping both genus-2 cycles healthy.

### Weil-Petersson Volume as Business Complexity

The Weil-Petersson volume of the moduli space grows super-exponentially with genus. For the AsymmFlow business:

- Genus 0 (a sphere): no cycles, trivial cash-only business
- Genus 1 (a torus): one credit cycle — the minimal viable trading business
- Genus 2 (our case): buy and sell cycles — the actual PH Trading complexity

Adding a third business cycle (e.g. maintenance contracts, MRR) would push the system to genus 3. The Weil-Petersson volume approximately doubles at each genus increment. This is the mathematical reason why adding recurring revenue to a trading business is nonlinear in complexity — it is not additive, it is a topological surgery that changes the genus of the business manifold.

## 5. Grothendieck's Abstraction: The ERP Scheme

### What Is the "Scheme"?

Grothendieck revolutionized algebraic geometry by noticing that a geometric space (a variety) is fully characterized by its ring of functions — by what you can measure on it, not by the space itself. He generalized this to schemes: spaces defined by their algebras.

For AsymmFlow, the analogous question is: what is the ring of functions on a business entity?

Every entity in AsymmFlow is characterized by:
1. **Identity fields**: who/what it is (id, customerId, supplierId)
2. **State fields**: where it is in its lifecycle (status, stage)
3. **Quantity fields**: how much (amountBhd, outstandingBhd, markupBps)
4. **Time fields**: when (createdAt, updatedAt, dueAt)
5. **Actor fields**: who acted (ownerId, createdBy, approvedBy)

This is the **scheme** — the abstract type from which all concrete entities are derived:

```typescript
type BusinessEntity = {
  // Identity
  id: u64;
  entityType: string;       // The type tag

  // Relationships (variable arity)
  relatedParties: { role: string, id: u64 }[];

  // State
  currentState: string;
  stateHistory: { state: string, changedAt: Timestamp, changedBy: Identity }[];

  // Financials (all in fils)
  obligations: { kind: string, amount: u64 }[];
  settlements: { kind: string, amount: u64 }[];

  // Temporal
  createdAt: Timestamp;
  updatedAt: Timestamp;
  terminalAt: Timestamp | null;
}
```

Every concrete table in AsymmFlow is a **point** (in Grothendieck's sense) of the spectrum of this type — a specific instantiation where most fields are fixed and only the variable parts remain free.

### The Topos of Business Objects

A **topos** is a category that behaves like the category of sets — it has all the constructions you need. The topos of business objects in AsymmFlow is the category where:

- Objects: all entity instances (all rows across all tables)
- Morphisms: the relationships between them (foreign key pointers AND business operations)
- Subobject classifier: the permission system (for each object, which roles can "see" it)

The STDB view system is literally the subobject classifier in action. A view for "Sales role sees opportunities" is a characteristic function `χ: Opportunity -> {0,1}` that is 1 for opportunities owned by the sales user and 0 otherwise. The collection of all such functions indexed by role IS the permission system, expressed as morphisms in the topos.

### The Etale Cohomology of RBAC

Etale cohomology detects "hidden symmetries" — structural patterns that are invisible locally but visible globally. For the permission system:

**H^0 (global sections)**: What every role can see. Answer: the dashboard, their own data. This is the "constant sheaf" of shared visibility.

**H^1 (first cohomology / obstructions)**: Where permissions create non-trivial cycles or contradictions. In PH Trading's RBAC:
- Sales creates opportunities and offers, but cannot approve them
- Manager can approve, but cannot create (in some configurations)
- This creates a **non-trivial 1-cocycle**: the approval workflow requires a handoff that cannot be composed from either role's permissions alone

This is why approval workflows always feel awkward in ERP systems — they are literally cohomological obstructions. The system has a non-trivial H^1. The costing sheet approval (markup < 20% needs manager, < 12% needs owner) is a concrete class in H^1(Biz, RBAC).

**The Grothendieck Insight for AsymmFlow**: The ONE abstraction that makes everything else fall into place is this:

**Model every business process as a sheaf over the time axis.**

A sheaf assigns data to each open set (time interval) consistently: the data on a smaller interval is the restriction of the data on a larger interval. Business entities are sheaves: an Invoice exists across a time interval, and at every moment within that interval, you can ask "what is the state of this invoice?" and get a consistent answer. The state machine transitions are the restriction maps.

When you frame it this way:
- Auditing is just computing global sections of the sheaf of activity events
- Reconciliation is checking that local sections (what the accountant recorded) glue to a consistent global section (what actually happened)
- The permission system is a sheaf of permitted operations over the identity space

The 156K LOC system never discovered this. The 12K LOC STDB system, with reducers as the only mutation path and subscriptions as the observation mechanism, accidentally implements it correctly.

## 6. The Open Source Module Decomposition

### The Kernel (Irreducible Core)

The kernel is what cannot be factored further — the piece every other module depends on.

```
@asymm/kernel
  - Identity management (Member table + join reducer)
  - Generic state machine (advance_pipeline reducer + transition tables)
  - Activity log (ActivityEvent table + log_event reducer)
  - Follow-up system (FollowUp table + create/complete reducers)
  - Permission model (role enum + permission check helper)
```

This is approximately 300 LOC. Everything else extends it.

### Universal Modules (Any Business Could Use)

These modules contain no PH Trading-specific knowledge. They are pure patterns.

**`@asymm/ledger`** (already built in experiment 001)
- Parties, obligations, settlements, splits
- The financial primitive: who owes what to whom
- ~400 LOC, zero domain knowledge

**`@asymm/pipeline`**
- The generic pipeline pattern: stages, transitions, terminal states
- Win probability as a configurable scoring function
- Follow-up scheduling
- ~500 LOC, takes a config object for domain specifics

**`@asymm/procurement`**
- The buy-side mirror of pipeline: PO, receipt, supplier invoice, payment
- Generic enough for any company that buys from suppliers

**`@asymm/intelligence`** (AI layer)

- Win probability calculator (competitor-aware, configurable)
- Translation/OCR procedure wrappers (Sarvam integration)
- ~600 LOC

### Module Composition: Does @asymm/ledger Plug Into @asymm/crm Without Modification?

Yes, with one constraint: the `obligation_id` foreign key pattern.

The ledger module tracks debts using `party_id` (who owes) and `obligation_id` (what for). The CRM module produces `Order` records. The connection point is: an `Order.id` becomes an `obligation_id` in the ledger when `create_invoice` is called.

This is a clean, one-directional dependency: `@asymm/crm` depends on `@asymm/ledger` (produces inputs to it). `@asymm/ledger` does not depend on `@asymm/crm` (knows nothing about opportunities or offers).

The composition diagram:

```
@asymm/kernel        (no dependencies)
     |
     +-- @asymm/ledger     (depends on kernel)
     |        |
     +-- @asymm/pipeline   (depends on kernel)
              |
              +-- @asymm/crm         (depends on pipeline + ledger)
              |
              +-- @asymm/procurement (depends on pipeline + ledger)
                           |
                           +-- @asymm/intelligence (depends on crm + procurement)
```

This is a DAG. No cycles. Each layer can be developed, tested, and published independently. A grocery store could use `@asymm/ledger` without `@asymm/crm`. A freelancer could use `@asymm/pipeline` without `@asymm/procurement`.

## 7. Concrete Recommendations (Synthesis)

Ranked by impact-to-effort ratio:

### Recommendation 1: One Generic State Machine Reducer (Highest Impact)

Replace all `update_po_status`, `update_offer_status`, `update_order_status`, `update_opportunity_stage` with one `advance_pipeline` reducer that takes `(entityType, entityId, newState)` and validates against a config object of valid transitions.

**Lines saved**: ~600
**Bugs eliminated**: All state machine duplication bugs
**Implementation time**: 2 hours

### Recommendation 2: Symmetric Obligation Handling

`Invoice` and `SupplierInvoice` are the same structure with `party_type: enum(Customer, Supplier)`. Same for `Payment` / `SupplierPayment`.

Merge them. One `Obligation` table. One `Settlement` table. Every reconciliation, outstanding balance, and aging report query becomes one query instead of two, unioned together.

**Lines saved**: ~400 in schema, ~800 in reducers
**Bugs eliminated**: The Phase 18 data reconciliation issues would not have existed
**Implementation time**: 3 hours

### Recommendation 3: Derived State Is Not Stored

Remove `outstanding_bhd` from Invoice. Compute it as `total_bhd - SUM(settlements)` in a STDB view. Remove `is_credit_blocked` from Customer. Compute it as `customer_grade == 'D' AND outstandingObligations > 0`.

This eliminates an entire class of reconciliation bugs — the Phase 18 work was entirely about correcting materialized state that had drifted from reality.

**Lines saved**: ~200 in reducers (no more manual outstanding updates)
**Bugs eliminated**: All "stale materialized column" bugs, which constitute the majority of data reconciliation work
**Implementation time**: 4 hours (view definitions + removing update logic)

### Recommendation 4: The Config Object Pattern

Extract all domain constants (grade thresholds, VAT rate, markup approval thresholds, margin targets) into a single `PHConfig` object in a constants file. No domain magic numbers should appear in reducer logic.

This is the "10 years of business knowledge in 30 lines" insight. It also makes the open source story clean: fork the config, keep the modules.

**Lines saved**: 0 (refactoring, not reduction)
**Value**: The entire system becomes domain-configurable. Any similar company can use it.
**Implementation time**: 1 hour

### Recommendation 5: The Activity Log as Projections, Not Tables

The `ActivityEvent` table should not be a permanent table. It should be an append-only event stream that feeds into views. The views compute the projections: "all events for invoice #42", "all events by user Alice this week".

This is event sourcing at the micro level. The underlying data model becomes the canonical truth, and the views become lens into it.

**Lines saved**: ~100
**Value**: Enables time-travel debugging (what was the state of the system at time T?)
**Implementation time**: 2 hours

## 8. The Minimum Description Length: Final Answer

The entire AsymmFlow ERP, including all business logic for PH Trading WLL, can be expressed as:

```
10 irreducible tables
+  8 generic reducers (parameterized by config)
+  1 config object (30 lines of domain knowledge)
+  5 STDB views (derived state)
+  ~4,000 LOC of Svelte UI (presentation)
= approximately 6,500 LOC total
```

This is 4% of the current 156,000 LOC codebase. The 156K LOC is not 24x more capable — it is 24x more redundant. The capability is in the business logic, which fits in ~2,500 LOC of substrate. The remaining ~4,000 LOC is user interface.

The 13x reduction claimed in the implementation plan (156K -> 12K) is conservative. The true irreducible minimum with disciplined abstraction is approximately 6,500 LOC.

## 9. The Connecting Thread

Mirzakhani counted geodesics on surfaces by understanding their moduli space. The insight: you do not need to study every geodesic individually. You study the space of all possible surfaces, and the counting follows from the geometry of that space.

Grothendieck algebraized geometry by replacing spaces with their algebras. The insight: a space is fully described by what you can measure on it, and these measurements form a ring.

Applied to AsymmFlow:

- You do not need to study every state machine individually. Study the space of valid transition tables, and every state machine follows as a configuration of that space.
- You do not need to build every obligation handler separately. Build the algebra of obligations (party, amount, outstanding, due date), and every invoice and PO follows as a ring element.
- The business intelligence layer is not a separate system. It is the sheaf cohomology of the obligation algebra over time — the global sections that make the local financial measurements consistent.

The 6,500 LOC system is not a simplification of AsymmFlow. It is the same system, written at the right level of abstraction. The 156,000 LOC was the system written before the right abstraction was found.
